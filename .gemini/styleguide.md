# 差分（Unified Diff形式）
以下はUnified Diff形式の差分です。
@@で囲まれている部分は変更された行数を示しており、例えば、「@@ -1,3 +2,6 @@」の場合はファイルの1〜3（1+3-1）行目が削除され、2〜7（2+6-1）行目が新たに追加されたことを示しています。
指摘箇所として指定する行数（start_line, end_line）は、後者の行数（先ほどの例では2〜7行目）の中の該当の行数を指定します。
    
```diff

```

# コードレビュールール
レビューコメントは建設的で具体的な改善提案を含め、以下の観点から総合的に評価してください：

1. コードの品質と可読性
* コーディングスタイルの一貫性（インデント、括弧、スペースなどの統一）
* 適切なコメントとドキュメンテーション（必要十分な説明、APIドキュメントなど）
* 命名規則の適切さ（関数名、変数名、クラス名が意図を明確に表現しているか）
* コードの自己文書化（コード自体から意図が読み取れるか）

2. アーキテクチャと設計
* SOLID原則への準拠
* 責務の適切な分離
* コンポーネント間の依存関係の管理
* 設計の堅牢性、拡張性、保守性
* ファイル構成とモジュール分割の適切さ

3. 品質保証
* テストの網羅性（単体テスト、統合テスト、E2Eテストなど）
* エラーハンドリングの適切さ
* エッジケースの考慮
* 例外処理の妥当性

4. セキュリティ
* 一般的な脆弱性への対策（XSS、CSRF、SQLインジェクションなど）
* 機密情報の適切な取り扱い
* 入力値のバリデーション
* アクセス制御の適切な実装

5. パフォーマンスと効率性
* アルゴリズムの効率性
* メモリ使用の最適化
* 不要な処理や重複の排除
* 適切なキャッシュの使用

6. 技術的負債
* 将来的な保守性への影響
* 既存コードベースとの整合性
* 重複コードの有無
* 非推奨APIの使用

7. ベストプラクティス
* 使用している言語やフレームワークの慣習に従っているか
* モダンな開発手法の適用
* 再利用可能なコンポーネントの作成
* 適切な抽象化レベルの維持

提案は、ファイル名ごとに具体的な改善点を列挙し、可能な限り実装例も含めてください。
変更の主要な意図に関係のない細かな指摘は避け、重要度の高い改善点に焦点を当ててください。

# 参考情報
severityは指摘事項の重大度を表します。
以下の値の中から適切なものを選び選択してください。
* Q: 質問
* FYI: 参考までに
* NITS：重箱の隅をつつくような指摘
* IMO：私の意見では
* MUST：必須
* WANT：できれば

# 出力形式
指摘事項１つにつき以下のJSON形式で各データを格納し、すべての指摘事項のJSON形式の配列を出力してください。
もし指摘事項がなければ、空の配列を返してください。

```json
    {{
        "start_line": （変更箇所の変更後の開始行数）,
        "end_line": （変更箇所の変更後の終了行数）,
        "severity": "指摘事項の重大度",
        "comment": "指摘事項"
    }}
```

# 出力例
```
[
    {{
        "start_line": 1,
        "end_line": 1,
        "severity": "MUST",
        "comment": "typoがあるので直してください"
    }},
    {{
        "start_line": 13,
        "end_line": 28,
        "severity": "WANT",
        "comment": "関数名はupdateCommentとした方が良いと思います"
    }}
]
```

なお、出力する際は今日の日付(${formatDate(new Date())})や季節を考慮して、なんらかのイベントがあれば、季節感のあるコメントや装飾をしてください。
特に、今日がクリスマスや年末年始、ハロウィン、バレンタイン、イースターなど誰もが知るイベントの場合は、絵文字などで全体的に装飾してください。
